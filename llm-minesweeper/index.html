Background

Why This Experiment?

Building on the success of the LLM Chess Evaluation — where Gemini 3.5 Flash demonstrated an estimated Elo of ~1900, competing with GPT-5.5 at a fraction of the cost — we wanted to explore whether these capabilities extend to spatial reasoning tasks on real mobile devices.

♟️

Prior Work: Chess Benchmark

Gemini 3.5 Flash was tested against different Elo levels of Stockfish and achieved an estimated rating of ~1900 Elo, making it competitive with GPT-5.5 while offering significantly lower cost. This strong performance in chess (a symbolic, well-structured game) motivated the question: can the same model handle a visual, spatial reasoning task like Minesweeper — especially when played through a real Android device UI?

Why uiautomator2?

ADB's built-in input tap command can send touch events, but it proved insufficient for complex gestures like pinch-to-zoom. The uiautomator2 Python library wraps Android's UIAutomator testing framework, providing reliable multi-touch gestures (pinch in/out), element selection, and more robust device interaction — making it the critical enabler for zoom controls in this experiment.

Methodology

Three Approaches Tested

The experiment iterated through three distinct architectures, each progressively giving the model more agency and better tools.

01
🔗

Dual-Agent System

A two-model pipeline: the first model (vision) analyzed the screen and extracted coordinates, then passed them to a second model (reasoning) to decide the next move. Complete failure — the reasoning agent couldn't interpret coordinates or understand the game state passed to it from the vision agent.

✕ Complete Failure
02
📋

Single Model + JSON Output

Following the chess harness pattern: a single vision model receives the full screen and outputs structured JSON describing its next move. The model could engage with the game but still struggled with accurately mapping visual positions to precise coordinates without spatial reference aids.

⚠ Partially Worked
03
🛠️

Single Model + Tool Calling

The final architecture: a single vision model with explicit tools (tap, long_press, zoom_in, zoom_out) — each with an optional reason parameter for the model to explain its logic — plus a coordinate grid overlay on every screenshot. This was the only approach that actually got the model playing — though it still lost every game.

✓ Best Approach (used)

Inside the Model's Mind

LLM Reasoning Traces

Each tool call includes an optional reason parameter where the model explains its Minesweeper logic. Game 1 provided no explanations, but Games 2 and 3 show surprisingly sophisticated deductive reasoning — even though the model still lost.

Game 1 — The model made all 18 moves without providing any reasoning (the reason parameter was not used). 0 of 18 tool calls included explanations.
Game 2 Move 1 → tap(550, 1150)
Starting tap near the center of the board to open up some space.
Game 2 Move 6 → tap(50, 1250)
C1, R13 is safe because C2, R13 is 1 and its only mine neighbor must be C2, R14 (which is the sole closed neighbor of C3, R13 = 1).
Game 2 Move 9 → tap(150, 1150)
C2, R12 is a mine because C1, R12 (1) only has C2, R12 as a covered neighbor. Thus C1, R11 (1) has its mine at C2, R12, which makes C1, R10 and C2, R10 safe.
Game 2 Move 15 → tap(250, 1750)
Using 1-1 pattern logic: C1,R18 (1) has covered neighbors in {C1,R17, C2,R17}, so the mine is there. C2,R18 (1) is adjacent to those two and also to C3,R18 and C3,R19. Thus, C3,R18 must be safe.
Game 2 Move 16 → tap(750, 1950) 💥
Using 1-1 pattern on C10, R19 (1) and C9, R19 (1), the closed neighbors of C10, R19 are a subset of C9, R19's closed neighbors, which means C8, R20 must be safe.
Game 3 Move 1 → tap(550, 1150)
Start the game by tapping a cell near the center of the grid.
Game 3 Move 4 → tap(300, 1400)
Row 14 Col 3 is safe because Row 13 Col 3 is 1 and its only mine neighbor is Row 12 Col 2 (which is a mine since Row 12 Col 3 is 2 and has only two unopened neighbors).
Game 3 Move 7 → tap(550, 1550)
Row 16, Col 5 is safe based on the deduction M(16,5) + M(16,7) = 0 from the adjacent 2s in Col 6.
Game 3 Move 8 → tap(450, 1550)
Row 16 Col 4 is safe because the set of unopened neighbors of the '2' at Row 15 Col 5 is a superset of the unopened neighbors of the '2' at Row 14 Col 5, meaning the extra neighbors must contain 0 mines.
Game 3 Move 12 → tap(150, 950) 💥
Col 2 Row 9 is safe because the adjacent '1' at Col 3 Row 10 is already satisfied by the mine at Col 2 Row 10.
🧠

The Reasoning Was Sound — The Perception Wasn't

The model demonstrates genuine Minesweeper strategy: subset-based mine deduction, 1-1 pattern recognition, constraint propagation, and counting-based logic. Games 2 and 3 provided reason fields in all 16 and 12 tool calls respectively — with increasingly sophisticated chain reasoning. The fatal errors weren't logical — they were perceptual. The model misidentified which cells were at which coordinates, or miscounted neighbors from the visual grid.

Time Breakdown

Each game step involves: screenshot capture (~0.5s) → grid overlay generation (~0.3s) → API call with image (~5-15s for vision model) → action execution (~1s) → animation wait (~2s). Including network latency, each step averages 10-20 seconds. The complete games ran for approximately 15-24 minutes each, totaling roughly 55 minutes of autonomous play across all three games.

Analysis

Key Takeaways

Three critical insights emerged from this experiment, each with significant implications for the future of AI-driven mobile automation.

Takeaway 01

Coding Agents Can Automate Mobile, Not Just Browsers

The fact that an LLM could connect to a real Android device via ADB/uiautomator2, capture screenshots, reason about the UI, and execute touch actions demonstrates that mobile automation through coding agents is viable . This exposes a dramatic new set of capabilities beyond browser automation.

Takeaway 02

Vision-Based Computer Use Is Still Fragile

Despite Gemini 3.5 Flash being a strong vision model, it struggled with Minesweeper — a game that's conceptually straightforward. Vision-based "computer use" through screenshots alone is still unreliable. Just as CDP (Chrome DevTools Protocol) was a breakthrough for browser agents over pure screenshot-based approaches, mobile agents will need structured data access (like a DOM-equivalent for mobile apps) to be reliable.

Takeaway 03

Data Format Is The Key Differentiator

When given Minesweeper as text/ASCII (like the chess experiment), the model would likely solve it easily. The failure isn't in reasoning — it's in visual spatial perception. This underscores that finding ways to extract structured data from mobile UIs (rather than relying on screenshots) will be the key enabler for practical mobile AI agents.

🔮

The Chess Paradox

Gemini 3.5 Flash achieves ~1900 Elo in chess (competing with GPT-5.5) when given symbolic / text-based board representations. Yet it cannot solve even basic Minesweeper when relying on visual perception alone. This reveals a fundamental asymmetry: LLMs are powerful symbolic reasoners but weak spatial perceivers . The bottleneck for mobile automation isn't intelligence — it's the input modality.

Coordinate Grid System

Every screenshot was overlaid with a 100px red coordinate grid before being sent to the model. This grid draws lines every 100 pixels, labels axis values along the edges, and plots coordinate text labels (e.g., "200,400") at every 100px intersection with a white dot marker. This was crucial — without it, the model hallucinated coordinates entirely and couldn't interact with the game at all.
